# Implementation Plan: Final Step towards Truly Channel-Agnostic Architecture

> [!NOTE]
> **System Optimization Strategy**: To reduce credit consumption during large refactorings, prioritize high-density turns. Batch multiple file edits using `multi_replace_file_content`, chain shell commands with `&&`, and perform exhaustive discovery tool-calls in parallel before implementation.

This document outlines the pending refactorings required to remove the remaining hardcoded channel references from the `apis-hub` core and transition to a fully modular, plug-and-play architecture.

## 1. Modularize Configuration Schemas

Currently, `ConfigSchemaRegistryService` and `ConfigManagerController` contain hardcoded definitions for Google and Facebook settings.

- [x] **Define `ConfigurableInterface`**: Update (or create) an interface that allows drivers to return their configuration schemas (JSON Schema format).
- [x] **Refactor `ConfigSchemaRegistryService`**: Change it to a dynamic registry that iterates through all available drivers to collect their requirements.
- [x] **Modularize Validation**: Move channel-specific configuration validation from `Helpers` to individual drivers.
- [ ] **Testing**: Run `phpunit` and verify `ConfigManager` UI displays dynamic schemas correctly.

## 2. Decouple Social Authentication

OAuth flows for Facebook and Google are currently handled by dedicated controllers (`FacebookAuthController`, `SocialAuthController`).

- [x] **Generic `OAuthDispatcherController`**: Create a unified controller that uses `DriverFactory` to retrieve the correct `AuthProvider` for a given channel.
- [x] **Refactor Callbacks**: Standardize state management so that the `redirect_uri` can be handled by a single endpoint that delegates to the driver's auth provider.
- [x] **Remove legacy views**: Replace `fb-login.html` with a dynamic button generated by the driver's metadata.
- [ ] **Testing**: Verify OAuth flow for at least two different providers (e.g. Meta and Google) using the same controller.

## 3. Modularize Demo Data Seeding

`SeedDemoDataCommand` is currently a monolithic file with dedicated methods for GSC, FB Marketing, and FB Organic.

- [ ] **Define `SeedableDriverInterface`**: Add a `seedDemoData(OutputInterface $output)` method to the standard driver contract.
- [ ] **Refactor `SeedDemoDataCommand`**:
  - Change the `execute` loop to iterate over `DriverFactory::getAvailableChannels()`.
  - Delegate the actual entity and metric generation to the driver.
- [ ] **Modularize Dummy Drivers**: Implement realistic simulation logic in `amazon-hub-driver`, `shopify-hub-driver`, etc.
- [ ] **Testing**: Run `app:seed-demo-data --channels=amazon` and verify data in DB.

## 4. Drive UI by Metadata

The reporting dashboards (`facebook-reports.html`, etc.) are static and channel-specific.

- [ ] **Define `DashboardMetadataInterface`**: Drivers should report which metrics, dimensions, and charts they support.
- [ ] **Generic Dashboard Engine**: Create a single `dashboard.html` that renders columns, filters, and widgets based on the driver's metadata response.
- [ ] **Dynamic Navigation**: Generate the sidebar menu dynamically based on enabled drivers and their supported entities.
- [ ] **Testing**: Swap channels in the UI and verify that available metrics/charts update automatically without page reload.

## 5. Refactor Channel-Specific Commands

Several commands are still tied to specific providers (e.g., `CacheFacebookEntitiesCommand`).

- [ ] **Generic `CacheEntitiesCommand`**: Implement a universal entity caching command that takes a `--channel` argument and uses the driver's internal discovery logic.
- [ ] **Refactor Coverage & Cleaning**: Standardize `ResetChannelCommand` and `CheckCoverageCommand` to handle any driver without internal `switch` cases.
- [ ] **Testing**: Run `app:cache-entities --channel=klaviyo` and verify cache entry generation.

## 6. Enum Evolution

- [ ] **Deprecate `Enums\Channel` as a Hardcoded List**:
  - Transition the system to treat `Channel` values as strings validated against the `DriverFactory` registry.
  - Use common identifiers (e.g., `meta`, `google`, `shopify`) as the primary keys.
- [ ] **Testing**: Add a "fake" driver via configuration and verify it appears in health checks and management lists without updating the enum.

## 7. Metrics Conversion Polish

`UniversalMetricConverter` still contains hardcoded Facebook-specific normalization logic.

- [ ] **Driver-Specific Normalizers**: Allow drivers to inject custom normalization closures into the `UniversalMetricConverter` or handle the raw data processing within the driver before passing to the converter.
- [ ] **Testing**: Verify conversion accuracy for a non-Facebook driver (e.g. Shopify) using the modular converter logic.
